# Geofencing Architecture  

Privacy-focused geofencing must balance OS limits, battery use, and end-to-end (E2E) location encryption.  In standard designs, **client-side geofencing** uses the mobile OS (Core Location on iOS, GeofencingClient on Android) to monitor zones, while **server-side geofencing** processes streamed locations in the cloud.  However, if locations are E2E-encrypted (server never sees raw coordinates), the server *cannot* naively check “enter/exit” for a geofence unless special cryptography is used.  In practice this means either doing geofence checks on-device or using advanced protocols.  For example, the NEXUS system shows that homomorphic encryption can let a server evaluate whether a user’s encrypted location lies inside an encrypted rectangle【20†L19-L27】【21†L79-L83】.  Such approaches reveal exact enter/exit results without ever decrypting the location or fence.  The tradeoff is high complexity: public-key homomorphic schemes or multi-party computation significantly increase CPU, memory and protocol overhead.  In contrast, pure client-side geofencing (no fancy crypto) is simple and efficient: the device decrypts or knows its location and zones and checks locally, preserving privacy but using battery and requiring client updates of fence data.  In summary, without sophisticated crypto protocols the **server cannot read encrypted locations to do geofencing**, so conventional systems do the evaluation on the client.  Homomorphic or private-set techniques *can* enable server-side checks in theory【20†L19-L27】, but at substantial implementation and performance cost.

## Client-side Geofencing APIs (iOS vs Android)  

Both iOS and Android provide built-in APIs for circular geofences. On **iOS**, you use Core Location’s region monitoring (e.g. `CLCircularRegion` with `CLLocationManager`) to define circular zones.  However, Apple strictly limits *how many* regions an app may monitor simultaneously to 20【13†L156-L164】【32†L69-L72】.  That is, “an app can register up to 20 regions at a time”【13†L156-L164】.  If you add a 21st region, the oldest one is dropped or an error is thrown.  iOS handles enter/exit callbacks via delegate methods when crossing these zones (after debouncing).  Note that iOS only supports circular (and beacon) regions natively – arbitrary polygons are not built in【32†L20-L24】.

On **Android**, the `com.google.android.gms.location.GeofencingClient` lets you define geofences similarly (latitude/longitude plus radius).  The Android API supports up to 100 active geofences per app (per device user)【9†L409-L413】.  The docs state: *“You can have multiple active geofences, with a limit of 100 per app, per device user.”*【9†L409-L413】.  Android’s API also allows a special “DWELL” transition type, which fires only after the user remains within a geofence for a configurable *loitering delay*【35†L889-L895】.  In summary, iOS Core Location: 20 circular regions max【13†L156-L164】; Android GeofencingClient: 100 max【9†L409-L413】.  

## Handling More Geofences than OS Limits  

Because 20 or 100 zones may be insufficient, apps use **dynamic/virtual geofencing** strategies.  A common pattern is *proximity-based activation*: only register the nearest zones to the user at any time.  For example, one can maintain a master list of hundreds of POIs but only “activate” (add) the ~20 closest ones on iOS as the user moves【6†L430-L437】.  As the device changes location, the app stops monitoring far-away regions and starts new regions near the user.  This way, the OS limit is never exceeded, but coverage is effectively large.  Another approach is hierarchical fences or “geofence tiers”.  For instance, create large-radius (coarse) circular fences that cover broad areas, and inside each large fence manage finer-grained sub-fences.  When the user crosses into a coarse region, the app then registers smaller child geofences for detailed monitoring【29†L155-L160】.  This **“coarse-to-fine” virtual geofencing** (also called proximity tiers) ensures only nearby zones consume resources【29†L155-L160】. In practice, production systems often mix both: use OS-native geofences for the highest-priority (e.g. nearest 20) zones, and fall back to server-based checks or dynamic updating for the rest.  

## Polygon vs Circular Geofences  

The native mobile geofencing APIs support **only circular fences**.  iOS `CLCircularRegion` and Android’s Geofence all take a center + radius【32†L20-L24】.  Polygonal geofences (arbitrary shapes) must be handled manually. In a polygon-approach, the client or server computes *point-in-polygon* math (e.g. ray-casting or winding-number algorithms) to test if the current location is inside a multi-vertex region【32†L26-L29】.  As one resource notes, “Most mobile OS APIs… natively support circular fences. Polygon geofences use a series of vertices… containment checks use the ray-casting algorithm or winding number method”【32†L20-L29】.  In other words, if you need an irregular zone (e.g. city block or custom delivery area), you can either (a) approximate it with multiple overlapping circles or (b) handle it yourself in code.  For (b), you typically register some enclosing circles to wake the app, then on location update run a custom polygon containment test.  This requires implementing the math yourself (there’s no built-in polygon geofence API on iOS/Android), but allows much more accurate zones【32†L26-L29】.  

## Enter/Exit and Dwell Detection Patterns  

Geofence monitoring always involves tracking an inside/outside state and transitions.  The basic events are *Enter* (cross from outside to inside) and *Exit* (inside to outside).  A third useful event is *Dwell*: the device remained inside a fence for a minimum duration.  The Android API explicitly supports a **DWELL transition**: if you set a loitering delay, the system will fire a dwell event only when the user has stayed within the geofence that long【35†L889-L895】.  As the docs explain, using `GEOFENCE_TRANSITION_DWELL` (with a loitering delay) “the dwelling alert is sent only when the user stops inside a geofence for a given period of time”【35†L889-L895】.  This helps suppress alerts when quickly passing through a zone.  iOS Core Location does not provide a built-in dwell.  Instead, developers typically implement dwell logic by recording the timestamp of an Enter event and then, if still inside after some threshold, manually handling it.  (Alternatively, Apple’s “Visit” API can sometimes be used for coarse “stop” detection, but it’s more oriented to general arrivals/departures over minutes【39†L1-L4】.)

iOS also imposes some built-in debouncing: the system may wait ~20 seconds after crossing before firing the event, to avoid jitter【6†L375-L379】.  In practice, geofence event code often enforces its own hysteresis or time checks.  For example, you might ignore a quick exit/enter pair as noise and only recognize a state change if it persists several seconds.  The goal is to avoid *flicker* near the boundary.  In summary, Android gives a convenient DWELL with loitering delay【35†L889-L895】, whereas on iOS you simulate dwell by adding timers or using higher-level visit APIs.

## Sharing Geofence Definitions in an E2E Circle  

When geofence definitions (zone coordinates and radii) must be shared among a group of users (a “circle”) without leaking them to the server, the solution is to treat the fence data itself as encrypted content. In practice, you would encrypt each geofence’s location data with the group’s shared keys and then distribute it via the server only as ciphertext. For example, one device can send the fence definitions over the end-to-end encrypted messaging channel or API that the circle uses; the server simply relays the encrypted JSON, and only clients decrypt it. This is analogous to how encrypted group chat delivers messages: the server never sees the plaintext. No standard library exists specifically for “encrypted geofences,” but common approaches include using per-recipient public-key encryption or a symmetric group key so that only circle members can decrypt the polygon or circle coordinates. In this way, the server *never learns the actual zone locations*. (We did not find a direct reference for this scenario, but it follows the usual pattern of encrypting any shared data end-to-end so untrusted servers only handle ciphertext.)
