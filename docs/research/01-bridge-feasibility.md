# 1. Bridge Feasibility per Service

- **Apple Find My:** Apple offers **no public API** for Find My (device or AirTag) location data. The only access is via the official apps or iCloud web interface. Enthusiasts have **reverse-engineered** it: e.g. the *FindMy.py* Python library logs in with an Apple ID (including 2FA) and queries Apple’s undocumented Find My service【60†L309-L317】. It decrypts “Find My” location reports (AirTags, iPhones, etc.) fetched from iCloud【60†L309-L317】. Authentication requires a full Apple ID login (with SMS/Trusted-Device 2FA)【60†L313-L315】. There’s no fixed polling limit documented; the library typically listens for Apple’s push notifications for updates, or polls periodically. Legally this violates Apple’s terms (since this use is unsupported), and it’s very fragile – any change in Apple’s protocol or web interface can break the solution. (In practice, Apple’s Find My updates are tightly controlled and not intended for third-party use, and workarounds often depend on staying logged into Apple’s own app to receive location pushes.) 

- **Google Maps Location Sharing:** Google also provides **no official API** to fetch friends’ shared locations. Attempts must use reverse-engineered hacks.  For example, the open-source `locationsharinglib` library logs in with Google account cookies and calls internal Google Maps endpoints to retrieve shared locations. A Google engineer confirms that “Shared locations are not available via [the] Google Maps API at the moment”【10†L0-L6】. In practice, one must authenticate with a valid Google account (web/mobile login) and maintain session cookies, then poll Google’s hidden APIs. Polling frequency is not published; one likely can poll at least every few seconds (like the Maps app refresh rate), but being too aggressive risks HTTP 429 throttling. This violates Google’s Terms of Service (unauthorized use of internal APIs) and is fragile – Google can and does change its endpoints or require new tokens without notice. 

- **Life360:** Life360 has **no public API**. The device apps communicate with Life360’s cloud using a private REST API. Community members have reverse-engineered it: they discovered that logging in with user credentials (phone or email + password) returns a bearer token from a CloudFront URL, which can be used to fetch circle and member data (locations, trips, etc.)【24†L48-L54】【24†L92-L93】. For example, one developer logged in, got a Bearer token, and polled each user’s location/trip data via the API. Poll intervals can be as frequent as ~15 seconds (some adapters set 60s by default and disallow below 15s to avoid limits【119†L1-L4】). However, Life360’s servers often **rate-limit** aggressive polling (HTTP 429/403 errors requiring backoff)【19†L271-L276】【119†L1-L4】. Legally, this is all *unsupported* by Life360; using it likely violates their Terms of Service. In fact, Life360 actively tries to block such use – Home Assistant dropped its built-in Life360 integration in 2024 because “Life360 and/or Cloudflare were actively blocking third party usage of their API”【19†L271-L276】. This reverse API method is thus quite **fragile**: updates to Life360’s backend (or stricter Cloudflare defenses) can break integrations, as seen by past outages and adapter fixes. 

- **Snapchat Snap Map:** Snapchat provides **no public API** for Snap Map or friend locations. The Snap Map feature is essentially closed. Some researchers have reversed bits of Snapchat’s private API (e.g. capturing Snap API calls), but Snapchat’s security tokens (signed by the official app) make this very difficult and unstable【40†L49-L52】【40†L57-L60】. Public Snap Map content (geotagged “Our Story” snaps) is actually viewable via `map.snapchat.com` without login【27†L84-L92】, but that only shows public snaps, not friends’ real-time locations. There is *no known* reliable open-source library to programmatically get private Snap Map data. Authentication would require emulating Snapchat’s mobile login (complex tokens)【40†L49-L52】. In short, **no API and no stable reverse hack** – any attempt would violate Snapchat’s terms and break whenever Snapchat updates its protocol. 

- **WhatsApp Live Location:** WhatsApp has **no public location-sharing API** beyond what’s built into chats. (WhatsApp Business API lets a business request/send a user’s location, but that’s different.) The only way to get a friend’s live location is to have them share it in a chat to your account. Community libraries like *Yowsup* and *whatsapp-web-reveng* reverse-engineer WhatsApp’s protocol (used by the Web client) and can receive incoming messages – including location updates embedded in messages. For example, Yowsup’s code shows handling a `<media type='location'>` message【45†L196-L204】. So a hacked client could log in with a phone number (via WhatsApp’s login protocol, e.g. scanning a QR or using tokens) and listen for incoming “location” messages. This is more pull than poll (it’s a persistent socket), so “poll rate” is as fast as you can handle incoming messages. However, this too **violates WhatsApp’s terms** (unofficial clients are banned) and is highly fragile (WhatsApp routinely changes its encryption/keys). No official or stable third-party API exists to continuously poll WhatsApp live location. 

# 2. E2E Encryption for High-Frequency Group Broadcast

We compare three protocols for fast group location messages (sender → group). In practice, both Signal (via its *Sender Keys* scheme) and Matrix’s Olm/Megolm use similar designs: each sender in the group has a symmetric ratchet key and a signing key, and broadcasts one encrypted packet to all recipients. 

- **Signal Protocol (Sender Keys variant):** For groups, Signal uses a *Sender Key* approach (derived from the WhatsApp “sender key” scheme). Essentially, each sender shares an ephemeral symmetric key with the group (exchanged via pairwise Double Ratchet sessions), and then ratchets/hashes that key for each message. To send an update, the sender hashes its group key to a new value, encrypts the message with it (e.g. AES), and attaches a signature. Recipients use the signature to authenticate and advance their copy of the key【72†L80-L88】【62†L146-L154】. Overhead per message is one symmetric encrypt + one signature/verify (much lighter than performing a fresh DH). The Signal sender-key approach provides confidentiality, integrity, **forward secrecy** and post-compromise security【62†L161-L164】, but it doesn’t automatically re-key on every group join/leave (the app periodically rotates keys or does a fresh pairwise exchange when needed). At high update rates (1–30 s), the ratchet overhead is minimal (just a hash and AES). Managing keys when members join/leave involves re-sharing a new sender key to the group via existing secure channels – this is O(N) work and can be awkward for many churns. Mobile CPU/battery impact is modest: AES+signature per message is cheap, and ratchet math is lightweight. Rust/Dart support: the Signal *libsignal* ecosystem has matured. For Rust there are crates (e.g. `double-ratchet-signal`), and for Dart the `libsignal-protocol-dart` package (with FFI to a native lib) exists【72†L80-L88】.

- **Olm/Megolm (Matrix):** Matrix uses *Olm* (Double Ratchet) for one-to-one chats and *Megolm* for groups. In Megolm, each sender in a room maintains an AES-based ratchet and an Ed25519 signing key. For each message they derive a new symmetric key (by hashing the previous) and encrypt the payload with AES-256-GCM, also signing it【68†L100-L109】. Other group members simply decrypt with that key. This means each message has one encryption + one Ed25519 signature. It provides sender-specific forward secrecy but **no backward secrecy**: if a sender’s key is compromised, all their future messages (until they re-key) can be decrypted【68†L69-L74】. Efficiency is good for high frequency: no expensive DH per message, only symmetric operations. On group membership changes, the sender typically *rotates* the session and shares the new key with others (again via pairwise Olm encrypted messages), which is O(N) work – a potential bottleneck for very dynamic groups. Battery use is similar to Sender Keys (AES+signature). Rust support: the `vodozemac` crate implements Olm/Megolm. Dart: the [matrix-dart-sdk] can incorporate it (it uses Rust libs under the hood). 

- **MLS (Messaging Layer Security, RFC 9420):** MLS is an IETF standard for large-group messaging. It uses a *tree-based ratchet*: members form a key tree so that a join/leave triggers efficient (log N) re-keying. For each message, MLS derives a fresh symmetric encryption key (from the current epoch) and encrypts the message once; the server fans it out. In practice, overhead per message is comparable to above (symmetric encryption plus an authentication tag). MLS is designed for both forward and backward secrecy automatically. The main advantage is group management: adding or removing members only requires updating a subset of the tree (O(log N) crypto work), instead of O(N). This suits larger, dynamic groups. At 2–50 people and sub-second updates, MLS’s overhead is minimal for sending: one AES encryption and (typically) a KDF on receive. However, existing MLS libraries are still maturing: *OpenMLS* is a Rust implementation that appears feature-complete (and has a Dart wrapper [openmls]), but there’s no widely used Dart client yet. 

**Summary Comparison:** All three use one symmetric encryption per message (plus authentication) so the per-message CPU cost is small even at sub-second rates. Signal’s sender-keys and Matrix’s Megolm are similar in efficiency (both use AES+signature); MLS also uses a symmetric cipher and possibly a MAC. Where they differ is in group membership handling: MLS is most scalable for frequent joins/leaves (using a tree), whereas Signal’s sender-keys and Megolm require new key distribution O(N) on re-key. In practice for 2–50 people, O(N) re-key is still manageable if roster changes are infrequent. Libraries exist: Signal’s Double Ratchet has Dart (libsignal) and Rust implementations; Matrix’s Olm/Megolm has Rust (vodozemac) and can be used from Dart; MLS has Rust (OpenMLS) and even a Dart wrapper. For an app sending 1–30 s location updates, any of these can achieve <1 ms per message encryption on modern phones. For simplicity and existing support, many would likely choose the Signal Protocol stack or MLS: Signal (via libsignal) is battle-tested in Dart/Flutter, and MLS (OpenMLS) is also promising. 

# 3. Background Location on iOS and Android (Flutter)

- **Permissions & App Store/Play approvals:**  
  - *iOS:* You must declare location permissions in `Info.plist` with `NSLocationWhenInUseUsageDescription` and `NSLocationAlwaysAndWhenInUseUsageDescription` (plus enable the "Location updates" background mode). By Apple policy, you **must** first request “When In Use” permission and only later request “Always” (iOS 13+ enforces this order)【79†L163-L170】. In practice, show good UI/UX explaining why you need always-on tracking. For App Store review, Apple requires a strong justification (e.g. navigation, friend tracking) and clear purpose strings【82†L292-L298】. Do not ask for Always immediately on launch; wait for a suitable in-app event and use a custom screens explaining why. Apple may reject or limit apps that abuse background tracking. iOS 14+ also introduces the ability for users to grant “Approximate” location (reduced accuracy) – if a user chooses this, you only get coarse updates, so you should detect and politely prompt them to allow full accuracy if needed.  
  - *Android:* Declare `ACCESS_FINE_LOCATION` (or coarse) and, for full-time background, also `ACCESS_BACKGROUND_LOCATION` in the manifest (Android 10+). Starting Android 11 (API 30), background location permission must be requested in a separate prompt after foreground permission. Google Play enforces a justification for background access via a policy questionnaire. On Android 12+ you also need `FOREGROUND_SERVICE_LOCATION` if running a foreground service for location【92†L148-L157】. In Flutter, the *geolocator* or *location* plugins will ask for these permissions in order. The Play Store requires that background location be truly needed; misuse can cause rejection.

- **OS-imposed update-frequency limits:**  
  - *iOS:* If your app has “location” background mode enabled, iOS will keep it alive for continuous GPS updates. However, iOS may throttle updates when the device is stationary or the app is suspended (it doesn’t promise exact frequency). There isn’t a documented “minimum interval” per se, but in practice Apple’s location service will send updates only when significant movement occurs unless you use the highest accuracy. There’s no strict OS limit on interval if you request `startUpdatingLocation`; expect on the order of ~10 sec or slower in background. iOS provides “Significant Location Change” API (wakes the app on ~500m moves) and “Visits” API to conserve battery, but those are coarser (minutes apart). Very high-frequency (1 s) updates in true background on iOS are unreliable unless the app stays active (foreground).  
  - *Android:* Without a foreground service, Android will aggressively limit location updates when the app is backgrounded (especially under Doze mode). If you run a **foreground service** (with a persistent notification), you can receive continuous updates. Android imposes no fixed lower-bound interval if you hold a WakeLock or use a continuous request, but the FusedLocationProvider’s fastest interval might be ~1 sec by policy. In Doze mode, a true foreground service should still get updates if high priority requests are used, but in practice update rate will slow (Android may batch them). WorkManager/AlarmManager can be used for periodic fetch, but not for 1–30 s. In short: use a foreground service + the FusedLocationProvider (in *flutter_background_geolocation* or similar) to get reliable updates in background. 

- **Flutter plugins (maintenance):** Active plugins include: **geolocator** (by Baseflow, updated regularly) for basic location (works in background with caveats), **location** (by the same author) which also supports background mode (requires `enableBackgroundMode()` on Android)【95†L91-L100】, and **flutter_background_geolocation** (TransistorSoft – commercial, but very robust with geofencing and optimized logic). The older **background_locator** (and background_locator_2) exist but are less actively maintained. Other helpers like **workmanager** can run tasks, but aren’t ideal for real-time streaming. 

- **Battery optimization / Doze:** On Android, to survive Doze/standby, you must use a **foreground service** with a notification (Flutter plugins above typically handle this). This tells the system your tracking is user-visible work, so it won’t suspend your app. You can also ask the user to exempt the app from battery optimizations. On iOS, background GPS inherently reduces accuracy if the app is not active; using the significant-change API or geofences conserves battery (Life360 uses this trick【100†L191-L197】). In general, combining coarse monitoring with bursts of high-accuracy updates (when movement is detected) saves power. 

- **What Life360/Find My do:** Life360’s iOS app reportedly immediately enables “significant-change” monitoring when backgrounded or terminated【100†L191-L197】, so it wakes on even subtle moves. It also runs in the foreground with a location service (likely as a “VoIP” or silent push trick). On Android, Life360 uses a foreground service. Apple’s Find My has privileged system APIs not available to third parties (it can use the device’s special permissions). Commercial apps often employ hidden tricks like silent push notifications to wake the app and then fetch location. 

- **Platform tricks:** 
  - *iOS:* Use **Significant-Change Location** service (`startMonitoringSignificantLocationChanges`), which can relaunch a terminated app on movement【100†L191-L197】. Use **geofence APIs** (wake app on entry/exit). In iOS 13+, there’s a “Background App Refresh”/Bluetooth peripheral advertisement hack (not guaranteed). 
  - *Android:* Use a **foreground service** with `startForeground()`. Use the `ACCESS_BACKGROUND_LOCATION` and `FOREGROUND_SERVICE_LOCATION` permissions. Some apps use WorkManager for periodic tasks, but for streaming location a persistent service is better. Also, Android 8+ requires a notification channel and request updates via the Google Play FusedLocationProvider for best results. 

# 4. Existing Open-Source Location Sharing Projects

- **OwnTracks:** A long-standing self-hosted tracker. *Architecture:* Mobile apps (iOS/Android) that *publish* location messages via MQTT or HTTP to your server【105†L43-L50】. A companion *Recorder* (Node.js-based) subscribes to MQTT (or accepts HTTP posts) and stores data in plaintext files, offering a built-in REST/WS API and simple web UI【110†L19-L27】【110†L47-L54】. *Tech stack:* iOS (Swift), Android (Java), server (Node/JS or optionally Docker image), storage in files (no DB required)【110†L19-L27】. *Good:* Fully private (you control the data), supports MQTT integrations (Home Assistant, Node-RED, etc.), lightweight. *Missing:* No built-in end-to-end encryption beyond TLS for MQTT/HTTP; UI is basic (map + history). It doesn’t “federate” between servers (you could subscribe multiple clients to the same MQTT topics if desired). *Encryption:* Data can be sent over TLS (MQTT over TLS or HTTPS). *Federation:* Only via MQTT bridging if manually set up. *Integration:* Many connectors (Home Assistant’s OwnTracks integration, mqttwarn, openHAB, etc.) can consume its MQTT feed. *Community:* Very active and mature (Recorder has ~1.2k stars, many contributors). 

- **Hauk:** An open-source live-sharing service. *Arch:* Backend is PHP + Memcached/Redis (no database)【112†L294-L302】. Users install the Hauk PHP code on a webserver and run a single Memcached instance; the Android app then creates ephemeral sharing sessions with that server. *Good:* Real-time sharing by visiting a short URL; very privacy-focused (location is never written to disk)【113†L281-L289】. *Missing:* Only Android client (no official iOS), no long-term history (data lives only in memory and is lost on reboot)【113†L281-L289】. Limited to individual sharing links – not suited for large-scale fleets. *Encryption:* Intended for HTTPS; plain HTTP would expose the share link (the docs warn strongly to use TLS)【113†L213-L221】. *Federation:* None (each Hauk instance is standalone). *Bridge:* No standard integrations; it’s mostly used ad-hoc for person-to-person shares. *Community:* Decent size (over 900 stars, active repo as of 2025). The author maintains it, and it’s packaged for Docker and Linux. 

- **Traccar:** A full-featured GPS tracking system (often used for vehicles). *Arch:* Java-based server (built on Netty), backed by SQL (supports MySQL, PostgreSQL, etc.)【115†L306-L314】. It implements **200+ GPS device protocols**, a rich REST API, plus web/mobile apps (“Traccar Manager” and “Traccar Client”) for Android/iOS【115†L306-L314】. *Good:* Enterprise-grade: real-time tracking, detailed history, reports, geofences, alerts, user/device management, etc.【115†L322-L330】. Very flexible (works with IoT trackers or phone GPS). *Missing:* More complex to set up; heavy-weight if you only need simple tracking. Its open-source UI is functional but not as polished as consumer apps. *Encryption:* Supports HTTPS and uses token-based API; one can secure it with TLS. *Federation:* No built-in federation – each server is standalone. *Bridge:* Has a plugin for Home Assistant and can feed into other systems via its REST API. *Community:* Large and active (7.1k stars on GitHub as of 2026【115†L385-L393】, frequent releases, commercial support available). 

- **GPSLogger:** (Android) *Arch:* A lightweight phone app that “logs” the device’s GPS to various formats (GPX, KML, CSV, etc.) and can **upload** data to custom URLs (FTP, email, Google Drive, Dropbox, OSM, etc.)【117†L283-L290】. *Good:* Very battery-efficient, offline-first, highly configurable. You can run it headless and aggregate your own logs. *Missing:* It’s not a sharing platform per se – there’s no built-in server or map view. You must supply your own backend (or use e.g. a cloud folder). No auto friend-sharing UI. *Encryption:* Uploads can use HTTPS if supported by target. No end-to-end user encryption. *Federation:* Not applicable. *Bridge:* It can send to any endpoint with a simple URL scheme, so you could “roll your own” server or integrate with cloud storage easily. *Community:* Popular among privacy enthusiasts (2.5k stars), actively updated by its developer.

- **OpenGTS:** (Not explicitly asked, but classic.) One of the first open GPS tracker backends (Java + SQL, Apache 2.0). It’s enterprise-grade but older (UI is dated). Features: tracking for fleets, geofences, reports. Its community is smaller now (largely overshadowed by Traccar). Encryption/Architecture is similar (runs on a servlet container with SSL).

**Summary:** These projects show that most open location services separate the *app* (on users’ phones) from a *server*.  Common architectures are: mobile app → HTTP/MQTT → backend (with optional database) → user-facing UI/API. OwnTracks and Hauk emphasize privacy and self-hosting; Traccar provides a full commercial-grade solution; GPSLogger is a personal logging tool. All use standard encryption (TLS) but few implement end-to-end crypto beyond that. None of them implement federated “peered servers” natively (data lives on your chosen server(s)). In practice, one would likely repurpose components (e.g. the OwnTracks apps or Recorder) rather than building entirely from scratch. 

**Sources:** Official docs and repositories for each service and protocol (cited above). For example, OwnTracks “uses MQTT or HTTP”【105†L43-L50】, Hauk’s FAQ explicitly states it holds all data in memory (Memcached)【113†L281-L289】, and Traccar’s README describes a Java server with SQL and REST API【115†L306-L314】. These illustrate each system’s design, strengths, and trade-offs. Each approach has trade-offs in update frequency, security, and resilience, which must be weighed for a new location-tracking solution.
