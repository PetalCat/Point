# Snap Map Reverse-Engineering Landscape

Snapchat’s **Snap Map** lets users share live location, but its internal mechanics are not publicly documented. In practice, the Snapchat app obtains device GPS and sends it over **HTTPS** (TLS on port 443) to Snapchat’s servers.  Early versions used a custom REST API (e.g. POSTs to `app.snapchat.com` with headers like `X-Snapchat-Client-Auth-Token`)【49†L123-L130】【50†L9-L17】.  Recent versions use Google’s Cronet HTTP/2 stack and a gRPC-based API (codenamed **MUSHROOM**) to send data【9†L262-L270】.  The protocol is protobuf-over-HTTP2: researchers note most “sensitive functionality” uses gRPC and that reversing it requires extracting Snapchat’s generated Protocol Buffer schemas from the app【9†L262-L270】.  All traffic is encrypted end-to-end and Snapchat pins its TLS certificates; the Android app verifies a hardcoded trust hash for Snapchat’s CA, requiring Frida-style unpinning to MITM【16†L542-L550】【9†L250-L259】.  

Snapchat does offer a limited **Places API** for developers (Lens Studio) to query nearby venues by lat/lng, but this is *not* user location sharing – it simply returns Snapchat Map “Places” data (venues) given coordinates【67†L207-L214】.  There is **no public API** for personal Snap Map data.  Internally, the app likely hits Snapchat’s private endpoints under Snap Inc. domains, authenticating with its login tokens.  In short, the mobile app uses HTTPS (port 443) with OAuth-like session tokens and Snap’s own auth headers【49†L123-L130】【50†L9-L17】; location updates are sent to Snap’s servers once every few minutes (or on movement) and Snap processes them even if you’re in “Ghost Mode” (per Snapchat’s own docs)【66†L344-L347】.  (As one Redditor confirmed: it “uses port 443 (which is https)” and likely checks certificates aggressively【64†L209-L217】.) 

## Published Research and Analysis

Several academic papers and security blog posts have examined Snapchat’s API and Snap Map.  Notably, Juhász *et al.* (2018) used Snap Map’s web interface (`map.snapchat.com`) to scrape public “Our Story” snaps via a WMTS tile API【26†L115-L122】.  Bellingcat’s investigation toolkit also notes the Snap Map web client and tools for OSINT.  Ghost Protocol (Elsevier 2021) describes how Snap Map can be exploited for surveillance.  In the infosec blogosphere, **Kevin Piacentini (2017)** detailed reverse-engineering of Snapchat’s login and token system, showing example HTTP requests and complex auth tokens in headers【49†L123-L130】.  **Hot3eed’s blog (2020)** digs into Snapchat’s binary obfuscation and confirms that each API request carried an `X-Snapchat-Client-Auth-Token` (and `req_token`) to thwart spoofing【50†L9-L17】.  Researchers at LRQA in 2018 and pyrosec in 2023 have published on Snapchat’s SSL pinning and anti-tamper: e.g. LRQA showed how the Android app loads its own keystore and how Frida can bypass TrustManager to accept a proxy’s certificate【16†L542-L550】【16†L569-L578】, while pyrosec noted Snapchat hashes its trusted cert fingerprints into the binary and needed custom Frida scripts to inject them【9†L250-L259】.  To our knowledge no full “open source” protobuf schema has been released; pyrosec reports that Snapchat compiles proto definitions into a custom lightweight (MessageNano) form, making static schema recovery hard【9†L262-L270】.  (However, casual OSINT tools scrape Snap Map via the web or by using the official web endpoints.) 

## Anti-Tampering and Certificate Pinning

The Snapchat app employs aggressive anti-reverse-engineering.  Security posts note that Snapchat’s Android client pins its SSL certificate: it loads a proprietary trust keystore in code, so standard MITM proxies (with user CA) are rejected【16†L542-L550】.  LRQA’s 2018 writeup finds that overriding `TrustManagerFactory.init` to use `null` (system default) or hooking `TrustManagerImpl.checkServerTrusted` via Frida can disable the pinning【16†L542-L550】【16†L569-L578】.  By 2023, Snapchat’s pinning grew more sophisticated: trust checks’ results are hashed into native code, requiring a two-step Frida approach (capture expected hashes, then inject them)【9†L250-L259】.  In practice, researchers have succeeded: once the pinning is bypassed, one can proxy Snapchat traffic through tools like mitmproxy or Burp by installing the proxy’s cert in the device.  (This is how some custom Snapchat bots or ‘tweaked’ clients work.)  Beyond TLS, Snapchat also obfuscates logic: hot3eed reports control-flow flattening and opaque predicates in native code, making static analysis very hard【50†L27-L36】. 

## Intercepting Snap Map Traffic

Because Snapchat’s map/location traffic goes over the same channels as the rest of the app, intercepting it requires bypassing the above defenses.  In the past, hobbyists set up an HTTPS proxy on the phone and successfully logged Snapchat location calls (as one user recalled doing with a custom iOS proxy)【64†L209-L217】.  Today, tools like Frida or Xposed (on Android) can disable pinning so that a proxy can see Snap Map gRPC calls in cleartext.  There are community scripts (e.g. pyrosec’s “snapchat-ssl-unpinning.js”) that automate this for Android Snapchat builds.  Once TLS is stripped, one can observe the HTTP/2 traffic.  A combination of Frida to remove anti-tamper and mitmproxy to log packets has been demonstrated by multiple researchers.  However, Snapchat’s APIs use persistent session tokens and anti-bot measures, so simply replaying captured requests usually fails without also simulating the token-generation logic. 

## Protobuf/API Schema (Leaked or Decompiled)

Snapchat’s internals use Protocol Buffers but Snapchat does *not* publish the `.proto` files.  According to pyrosec, the Snap app contains compiled protobuf stubs (MessageNano) without human-readable fields, so recovering the schema requires reverse-engineering the DEX/bytecode【9†L262-L270】.  Some OSS projects (e.g. on GitHub) have partially reverse-engineered Snapchat’s “private API” and share unofficial endpoints/methods, but these are fragile.  We found no official leaked schema dumps.  In practice, the best clue is observing traffic after unpinning.  Tools like `grpcurl` could in theory hit the live endpoints, but Snapchat likely uses authentication tokens and nonces that must be obtained from the client.  In summary, Snapchat’s gRPC API (“MUSHROOM”) schema is not publicly documented – only hints exist in decrypted traffic and partial decompilations【9†L262-L270】. 

## Legal and Privacy Considerations

Snapchat location data is **private and protected**.  Your own location (shared via Snap Map) is accessible to you and to the friends/groups you explicitly allow, but reading it still passes through Snapchat’s service.  Legally, pulling data from your *own* account is generally permissible (it’s your personal data), but intercepting or scraping someone else’s location without consent can violate privacy laws (e.g. wiretap or data privacy statutes) and Snapchat’s Terms of Service.  Notably, Snapchat’s Safety Hub and case law indicate that Snap Map data is treated as user content: law enforcement can subpoena it with a warrant【61†L171-L179】【61†L180-L188】.  In 2025 a law firm noted that “location data, including Snap Map activity,” can be obtained as evidence【61†L180-L188】.  This implies Snapchat retains location logs for at least 24 hours (since Map Snaps expire after a day).  In short: accessing your own Snap Map location via official means (the app) is fine, but any unauthorized scraping of *others’* locations likely violates privacy policies or laws.  (For OSINT, one may legally view **public** Snap Map stories or posts, but not private friend locations.)

## Alternative Approaches on the Device

One might ask if a companion app on the same phone could read Snapchat’s location data directly.  In general, modern mobile OSes **sandbox apps**: Snapchat’s private storage and memory are inaccessible to another app unless the device is rooted/jailbroken.  Snapchat does not broadcast its location internally – it simply reads from the OS location service and sends that data to its server.  A separate app can request GPS itself, but it won’t know when Snapchat decides to report its location (nor can it intercept Snapchat’s in-app data).  Snap Map location also isn’t stored in a shared database: it’s ephemeral.  (Snap’s documentation notes that enabling Snap Map requires OS location permission, and even if you hide on Ghost Mode, “Snapchat still processes location data”【66†L344-L347】. This suggests location is kept server-side, not in a readable file on the phone.)  In short, without breaking the OS sandbox or Snapchat’s own app, a companion app cannot siphon Snap Map data from Snapchat’s private storage or broadcasts.
