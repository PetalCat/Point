# MLS (OpenMLS) Implementation for Mobile Apps

- **Maturity:** OpenMLS (Rust implementation of RFC 9420) has matured significantly. It is now at version 0.8.x (2024–2025) with full RFC9420 support; the project’s docs list core functions as “Done” (group creation/join, member add/remove, messaging, commits, etc.)【65†L93-L100】. Major companies are already using MLS: for example, Cryspen reports OpenMLS is used by **Wire** and **Matrix**【6†L225-L233】 (Wire even advertises itself as the first collaboration platform “completely secured by MLS”). Other projects (e.g. XMTP) have announced MLS-based group chat【47†L27-L35】. In summary, OpenMLS is no longer purely experimental – its features are largely implemented and it’s being field-tested in real systems – but it is still evolving with active development and benchmarking.

- **API for Groups & Messages:** The OpenMLS API (as exposed by its Rust library and Dart bindings) provides high-level functions for group and member management. For example, one can call `createGroup(...)` or `joinGroupFromWelcome(...)` to create or join a group, and `addMembers(...)`/`removeMembers(...)` to change membership【65†L108-L116】. After group setup, sending a message involves calling `createMessage` (which produces an encrypted MLS message) and recipients use `processMessage` to decrypt and verify it【65†L131-L134】. In practice, OpenMLS’s Dart library (`openmls` package) exposes these operations via Flutter Rust Bridge, and documentation shows calls like `MlsEngine.createGroup(...)`, `engine.addMembers(...)`, `engine.createMessage(...)`, etc. All major MLS operations (proposing and committing membership changes, external join, etc.) are implemented【65†L93-L100】. In short, the API exposes MLS primitives (KeyPackages, Welcome messages, Commits, etc.) in user-friendly functions for creating groups, adding/removing members, and encrypting/decrypting group messages【65†L108-L116】【65†L131-L134】.

- **Epoch-Change (Commit) Overhead:** MLS update commits are efficient. Benchmarks from the OpenMLS team show update costs grow sub-linearly with group size (O(log n) in the “fully updated” case)【31†L118-L127】. For instance, a 300-member group only requires ~9 HPKE encryptions to send an update (versus 299 in the worst-case “bare” tree)【31†L136-L144】. In practical terms, end-to-end timing experiments report on the order of single-digit milliseconds per update even for large groups.  Thus, doing commits on every message (say once per second) adds only a few milliseconds of CPU work on modern hardware.  In summary, at message frequencies of 1–30 s, the cost of epoch changes is negligible compared to the interval – most of the work remains cryptographic and is small relative to 1 s. 

- **Dart/Flutter FFI Bindings:** An official Dart package `openmls` provides Flutter integration via [flutter_rust_bridge](https://pub.dev/packages/flutter_rust_bridge)【65†L93-L100】.  This means OpenMLS is already callable from Dart/Flutter. Flutter Rust Bridge (FRB) is a mature, Flutter-favorite binding generator【72†L135-L138】 that supports complex use cases. It automatically handles **async Rust functions** (you can `async fn` in Rust and `await` it in Dart)【50†L64-L72】, and even allows Rust to emit event streams that Dart listens to via a `Stream`【51†L43-L51】. FRB can translate many complex types (structs, enums, vectors, maps, Results, etc.) between Rust and Dart with minimal boilerplate【72†L165-L173】. In practice, the OpenMLS Dart package uses FRB to bridge into the Rust library. No major “gotchas” are reported: FRB abstracts away most FFI details. (As always with Dart FFI, one must not share non-FFI-safe Dart objects into Rust and be mindful of isolate boundaries, but FRB’s codegen handles the common cases.) In short, stable Flutter bindings for OpenMLS already exist via FRB, and they handle async calls and streaming updates smoothly【50†L64-L72】【51†L43-L51】.

- **Key/State Storage:** Each MLS client must **persist its group state and key schedule position**.  Concretely, a client stores its *MlsGroup* state (group ID, current epoch, tree, member list, its own leaf node private key and credential, pending proposals, etc.) in secure storage. The MLS spec explicitly requires persisting the ratchet generation counters (so you don’t reuse a key/nonce)【36†L2025-L2030】. OpenMLS provides a `StorageProvider` interface; the default Rust engine uses an encrypted SQLCipher (on mobile) or IndexedDB (on web) to store this state【44†L27-L35】.  As a result, all secret material (private keys, path secrets, sender-data secrets, etc.) are written to this secure storage, and OpenMLS regularly deletes old secrets to maintain forward secrecy【44†L33-L41】. In summary, each client must keep its **entire MLS state (ratchet tree, credentials, counters, group context)** persisted, so that after a reboot or wake-up it can continue correctly. OpenMLS handles this via its `MlsEngine` and encrypted storage layer【44†L27-L35】, but the developer should ensure the encryption key (e.g. 32-byte AES key) is itself kept safe (e.g. in the OS keystore)【44†L33-L41】.

- **In-Production Use:** Yes – MLS (and likely OpenMLS) is already in use. Cryspen reports that “OpenMLS [is] used by multiple companies including Wire and Matrix”【6†L225-L233】.  Wire’s marketing confirms it has baked MLS into its platform.  The XMTP messaging protocol announced it added MLS-based group chat【47†L27-L35】. Industry insiders also cite Cisco/WebEx and others experimenting with MLS in production.  (Discord’s voice encryption is already built on a group key mechanism inspired by MLS.)  In short, while MLS is new, it is being adopted by real-world apps: major encrypted messaging and collaboration platforms are piloting or rolling it out. OpenMLS, being one of the main implementations, is at the core of this early deployment.

- **Signal Sender-Keys vs MLS:** An alternative approach is to use Signal’s Sender Key group encryption. In Flutter, one could use the `libsignal` Rust binding (there is a Dart package) that implements sender-key groups【49†L95-L104】. Sender-Key groups are simpler and proven (used by WhatsApp/Signal), so they carry fewer protocol-unknowns.  A disadvantage is they give weaker *post-compromise security* (if a group key is compromised, future keys can be recovered by the attacker until a rekey). MLS is newer and more complex but offers stronger forward- and post-compromise secrecy and better scalability (cryptographic operations grow as O(log n))【31†L118-L127】.  From an engineering standpoint, both options require Rust FFI (the libsignal Dart package uses FRB too). Using libsignal’s sender-keys may feel more “known” since the Signal protocol is battle-tested and well-documented. MLS (OpenMLS) has more unknowns (spec is fresher), but is now feature-complete and formally reviewed.  In summary: **Sender-Keys (libsignal) has less novelty risk and simpler semantics, but MLS (OpenMLS) promises stronger security guarantees.** The choice depends on priorities: if you need maximum security and scalability, MLS is worth it; if you want the most battle-tested solution today, sender-keys may have fewer conceptual unknowns. 

# Rust + Flutter FFI Architecture

- **flutter_rust_bridge (FRB):** The recommended bridge is [flutter_rust_bridge](https://github.com/fzyzcjy/flutter_rust_bridge) (officially **Flutter Favorite**【72†L135-L138】).  FRB auto-generates the Dart/Rust glue code. It **supports async Rust functions** out-of-the-box (you write `async fn` in Rust and `await` it in Dart)【50†L64-L72】. It even supports **streams of events**: Rust can send events into a `StreamSink` and Dart can listen on a `Stream`【51†L43-L51】. Complex types (Rust structs, enums, arrays, etc.) can be passed back and forth, with FRB deriving the necessary serialization【72†L165-L173】. In practice, you mark up your Rust API functions (in an `api.rs` module) and run `flutter_rust_bridge_codegen`, which scans for extern functions and generates Dart bindings. FRB supports returning Rust errors as Dart exceptions, passing vectors/strings/byte buffers, and mapping Rust `Result`, `Option`, and even nested collections to Dart equivalents. The v2 FRB documentation and changelog note that it can handle arbitrary Rust types, closures, and even call Dart from Rust【72†L165-L173】. In short, FRB is robust: it handles async, streams, and rich data types with minimal manual work【50†L64-L72】【72†L165-L173】. 

- **Shared Rust Crate Structure:** A common pattern is to put **core logic in a shared Rust library** and have both the server and Flutter app depend on it. For example, you can set up a Cargo workspace with a `shared_core` crate (containing crypto routines, serialization schemas, protocol logic) and two targets: one binary for the server, one for the Flutter bridge. The Flutter app’s Rust side would include an `api.rs` that calls into `shared_core`. (Similarly, the server directly uses `shared_core`.) This way, the business logic is written once. As one source explains: *“Make a third project, a library, that both other projects depend on”*【15†L217-L224】. In practice with FRB, you might place your shared code in `lib/` and have `src/api/` (the FRB part) re-export or call into it. The build pipeline can compile this shared crate for both mobile and server targets. The key is: **don’t duplicate crypto code**; use a single Rust crate with features or workspace members for each environment.

- **FFI Call Performance:** Dart’s FFI calls are very fast. Once a native library is loaded, each call is roughly as fast as a typical C dynamic call (on the order of micro- or nanoseconds)【54†L85-L93】.  In fact, benchmarks show Dart FFI overhead is only ~100 ns per call【54†L85-L93】. This is **orders of magnitude faster** than Flutter’s platform channels. Thus, decrypting or encrypting at 1–30 s intervals will not be dominated by FFI overhead – the overhead is negligible. (The main cost will be the cryptographic computation itself.) However, one must avoid *very* chatty interfaces (e.g. thousands of tiny calls per second), because the fixed FFI crossing cost would accumulate. In our use-case (say one call per message), this is no issue. In short: the FFI overhead is minimal (100 ns to 1 μs per call)【54†L85-L93】, so performance hinges on the native code speed, not the call overhead. 

- **Build/Cross-Compilation Pipeline:**  To support all Flutter targets, you generally compile your Rust crate for each platform:

  - **Android:** Use [`cargo-ndk`](https://github.com/bbqlinux/cargo-ndk) or manual NDK to build shared libraries (`libmylib.so`) for each ABI (armeabi-v7a, arm64-v8a, x86, x86_64). Place the resulting `.so` files in `android/app/src/main/jniLibs/<ABI>/`. Flutter then bundles them. (Some templates automate this.) 

  - **iOS/macOS:** Use [`cargo-lipo`](https://github.com/TimNN/cargo-lipo) or [`cargo-xcode`](https://github.com/TimNN/cargo-xcode) to build a universal static library (`libmylib.a`) or XCFramework. Then include that in your Xcode Runner target. You can use `DynamicLibrary.process()` in Dart to load the library at runtime, or link it. In short: compile Rust with `-target aarch64-apple-ios` and `-target x86_64-apple-ios`, then bundle the results.

  - **Web (WASM):** For Flutter Web, you compile your Rust crate to WebAssembly. For example, using `wasm-pack build -t no-modules` with the appropriate features (atomics, bulk-memory) as FRB guides recommend【74†L102-L110】. This produces a `.wasm` module and JS loader that you put under `web/pkg/`. You must also serve the web app with COOP/COEP headers (“Cross-Origin-Embedder-Policy: require-corp” etc.) as shown in FRB docs【74†L135-L142】. Once set up, Flutter Web can load the WASM module to call your Rust code in the browser.

  - **Desktop (Linux/macOS/Windows):** Just compile for the host platform (`x86_64-unknown-linux-gnu`, etc.) as usual. 

  In practice, CI scripts or build.rs can loop over targets. The key point is: one Rust crate can be compiled multiple times with different targets. Tools like `cargo-ndk`, `cargo-lipo`, and `cargo-xcode` simplify the process. A summary from a Flutter+Rust guide says: *“For Android: build .so for each ABI under jniLibs; for iOS: produce an XCFramework (.a) and add to the app; tools like cargo-ndk/cargo-lipo are recommended”*【56†L138-L146】.  Flutter Rust Bridge’s codegen or example projects often include scripts that do these steps automatically.

- **Real-World Flutter+Rust Examples:** The pattern of embedding native code is well-proven. Many large apps use a compiled core (usually C++/Rust) for heavy logic.  For instance, Telegram and Signal embed C++ crypto engines in their mobile apps【56†L99-L107】.  Discord uses C++ for audio/video in Flutter clients.  Embedding Rust in Flutter is conceptually similar.  Open-source examples of Flutter apps using Rust include small demos and hobby projects, but major companies have not widely publicized it yet.  However, the approach is essentially the same as in those big apps: a smooth Flutter UI with a fast native (Rust) core. As one summary notes: “Companies like Telegram, Signal, Discord and Brave rely on a compiled, cross-platform core for performance-critical features【56†L129-L134】.” We can adapt that strategy with Rust.  (In fact, even if not using Flutter, these examples show that high-performance native logic is standard in messaging apps.) We may not have a famous Flutter×Rust production app to name, but the architecture is proven at scale.

- **Rust FFI vs. Pure-Dart Crypto:** Using Rust (via FFI) usually gives better performance and control than pure Dart crypto libraries. For example, one benchmark showed a Rust AES-GCM implementation achieving **≈5× the throughput** of Dart’s (in file encryption)【75†L166-L174】, largely due to native AES instructions and no Dart-GC overhead. Another consideration is security: Dart’s VM has a garbage collector, so secret keys in Dart may linger in memory unpredictably. In one Flutter example, developers moved all crypto to Rust because *“the Dart GC was a bottleneck – keys could not be wiped easily and remained in memory”*【23†L88-L97】.  That said, pure-Dart libraries (like the [cryptography](https://pub.dev/packages/cryptography) or [encrypt](https://pub.dev/packages/encrypt) packages) are far simpler to integrate (no FFI), and for moderate workloads their speed is often acceptable on mobile. The trade-offs are: **Rust FFI** gives raw speed, hardware acceleration, and deterministic key wiping, but adds build complexity and manual memory management (you must free any allocated buffers yourself). For instance, one Rust+Flutter example warns “Things to watch out for: manual memory management, struct alignment, panics become Dart crashes”【75†L173-L179】. Meanwhile, **Dart crypto** is GC-managed and easier to use, but slower and less low-level (no direct access to AES-NI, no guaranteed key zeroing).  In summary, Rust+FFI is “worth it” if you need top performance or strict security guarantees. If you only need basic encryption and want maximum simplicity, pure Dart crypto might suffice. (For example, encrypting short messages via Dart may be fine, but bulk file encryption often wins with Rust【75†L166-L174】.)
